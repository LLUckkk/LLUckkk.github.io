P5第三题：lwrev

取读到DM数据的后四位0扩展得到目标寄存器，再把rt本身的编号0扩展写入这个目标寄存器。

我的做法仅仅只是在controller中添加了信号并且在最后把目标寄存器和写入寄存器的数据修改了。仿佛一个傻子，根本不知道流水线在干什么

添加条件访存指令究竟需要注意什么

总共三种类型，条件均为从DM中取值进行判断：

条件成立 ：将DM/其他值写入A号寄存器；条件不成立：将某个值写入B号寄存器

  1. 宏文件中添加对应的op码识别信号，因为是load型指令，所以也要在load后面添加，如果是半字或字节的话在对应的字宽信号后面添加指令信号，如果写ra寄存器的话也要加一下ra信号，同时因为有条件，所以需要在compare中添加对应的compare_op

    op -> load -> (byte/half) -> (ra) -> com_op

  2. controller中，RegWrite修改成如果条件成立的话写入的寄存器；添加RW信号，添加com_op，添加check_2

    RegWrite -> RW -> com_op -> check_2

  3. hazard中对应以下的板子，也即遵循的逻辑是，只要前面指令操作的是该指令可能写的寄存器，就暂停。对于转发信号,W级之前均不转发，只在W级进行转发，情况2应该改成E/M级要写的寄存器，或者是这个指令如果条件满足的话需要写入的寄存器。

  1. 在compare中实现对应的比较行为。

  2. 对于外部，condition信号是W级compare模块产生的，需要修改的是W阶段回写的寄存器（因为该寄存器只在W阶段才知道是什么，所以仅仅修改W阶段，如果其他阶段也能进行判断的话，其他阶段应该也需要修改）和回写的数据（同前），

    check && condition -> WD -> writeReg_W

条件成立：将DM/其他值写入A号寄存器；条件不成立：不写入

与上一条相同，不写入寄存器的话就在最后修正目标寄存器的时候把不符合条件的写成0号寄存器，注意此时不要再搞那个com_write了，过于复杂，可能会和条件写混在一起，所以不要再写了。

写入的目标寄存器跟从DM中读出的数据有关

肯定需要暂停。注意暂停和转发即可，以及最后目标寄存器和写入数据的修改。

本质上一个类型，需要明确以下几点：

  • 只有到M级才最快知道写入目标是什么。因为condition的判断需要用到DM中的值。
  • 因此导致转发和阻塞都会受到影响。因为转发和阻塞的判断条件是这一级要用的寄存器在后面的指令中被修改了，但是还没有写入寄存器堆。但是现在究竟哪个寄存器会被修改还并不确定。
  • 可以看到，第三条是肯定需要暂停的。

解决办法：对stall信号的生成逻辑修改.

如果D级的指令要读寄存器，而且后面的新指令可能要写这个寄存器，就执行stall

因此需要修改能够触发stall的条件，注意，此处只判断D级指令。

  1. 首先在controller中添加check信号判断是这个特殊的指令
  2. 接着在原有address判断的基础上，添加”如果是这个特殊指令，那么要用的寄存器可以等于两种情况。”

条件写入中寄存器的地址务必在M级进行修正，否则会导致错误转发。看一个例子。

P5第二题：b什么玩意

  • 题目记不太清了，暂且称为bxx，

无条件写入pc+8，如果有符号的rs和rt的最小值是偶数则跳转，否则清空延迟槽。

此外题上描述的跳转也很值得思考。

解释一下指令手册中的pc

原来的beq，操作手册上说的是PC+4+offset||00，但是题上描述的是PC+offset||00。

看了眼评论区，是自己太蠢了没有补pre教程：

  • 操作手册中标注了I和I+1，I指当前指令的指令时间，I+1指的是当前指令的后一条指令的指令时间。

  • 指令时间：PC指向了哪条指令的地址，现在就是哪条指令的指令时间。

  • 对于操作手册来说，这里的PC永远指的是当前F级的PC。

对于题目来说，题目中第一句：

是在I+1标签后的，PC指的是F_PC。具体来说，此时bxx指令已经到D级了，算偏移量的时候，用的是F级的PC。但是由于流水线天然具有延迟槽的属性，所以这里的PC，实际上就是之前单周期说的PC+4。也就是说，这里的pc和PC_D + 4是一样的。

也即PC_F = PC_D + 4

题目第二句：

是在I标签之后的，也就是bxx这个指令在F级的时候，那么这个PC指的就是bxx所在的PC，也就是一直传下去的PC，因此和jal也没有区别。

一句话总结：新指令的跳转方式和原来的相同，都是PC_D+4+offset||00。

添加条件跳转指令究竟需要注意什么

条件跳转指令的类型一般有以下几种：

条件跳转加无条件写

  1. 宏定义中增加信号，包括op码识别信号，branch后添加信号，如果是两位的比较在branchTwo添加信号，如果是无条件写31寄存器，再加一个ra信号，最后务必注意添加compare_op信号

    op -> branch -> (branchTwo) -> (ra) -> (bxx_op)

  2. controller中注意RegWrite，如果要写的寄存器是31号则不用管，如果写的是rt寄存器，不用管；如果写的是rs寄存器，需要在冒号后添加一种特殊情况。即“如果是这个指令，那么就选这个寄存器。” RW中同样也需要添加这个指令。com_op 务必记得加！！，就是在com_op中选择“如果是这条指令，那么com_op就为对应的op”

    RW -> RegWrite -> com_op

  3. 补充compare里面的比较方式

  4. 别的就无需在添加了，需要流水的控制信号没有，因为D阶段就已经用过了，因为是无条件写所以也不用传condition信号

条件跳转加条件连接

  1. 宏定义中增加信号，包括op码识别信号，branch后添加信号，如果是两位的比较在branchTwo添加信号，如果是无条件写31寄存器，再加一个ra信号，最后务必注意添加compare_op信号

    op -> branch -> (branchTwo) -> (ra) -> (bxx_op)

  2. controller中注意RegWrite，如果要写的寄存器是31号则不用管，如果写的是rt寄存器，不用管；如果写的是rs寄存器，需要在冒号后添加一种特殊情况。即“如果是这个指令，那么就选这个寄存器。” RW中同样也需要添加这个指令。com_op 务必记得加！！，就是在com_op中选择“如果是这条指令，那么com_op就为对应的op”，除此之外，因为是条件写，所以在com_write中也需要添加这个信号。

    RW -> RegWrite -> com_op -> com_write

  3. 在寄存器中需要流水compare_condition信号，同时需要修改EMW级的RW的确认方式，按照学长教的老方法来确定。这决定了是否要进行转发，因为也有可能因为条件不满足根本就没写寄存器，这时候就不用转发。

    EMW级的RW信号 ，流水compare_condition信号

  4. compare中增加对应的比较方式

条件跳转加条件/无条件连接加不跳转时清空延迟槽

  1. 宏定义中增加信号，包括op码识别信号，branch后添加信号，如果是两位的比较在branchTwo添加信号，如果是无条件写31寄存器，再加一个ra信号，最后务必注意添加compare_op信号

    op -> branch -> (branchTwo) -> (ra) -> (bxx_op)

  2. controller中注意RegWrite，如果要写的寄存器是31号则不用管，如果写的是rt寄存器，不用管；如果写的是rs寄存器，需要在冒号后添加一种特殊情况。即“如果是这个指令，那么就选这个寄存器。” RW中同样也需要添加这个指令。com_op 务必记得加！！，就是在com_op中选择“如果是这条指令，那么com_op就为对应的op”，除此之外，如果是条件写，在com_write中也需要添加这个信号。最后，因为涉及清空延迟槽，所以在check_1中添加该信号。

    RW -> RegWrite -> com_op -> （com_write）-> check_D

  3. 如果是条件写的话参照条件写的第三条，无条件写的话参照第一种类型

  4. 清空延迟槽，在D级就可以进行了，单独设置一个信号。同时在D级寄存器的reset输入端口添加该信号，注意此时不能是阻塞的情况，如果阻塞是不能清空延迟槽的

关于阻塞

一般阻塞一个周期，在该周期的下一个周期的时钟上升沿，PC和PCD都不动，也就是PC冻结，D级指令不动，E级继续往前流水，在下一个周期，数据转发过去了，做出了正确的行动，再往前流水。在阻塞结束之后一切都会自动恢复正常，无需纠结。

做了一些题，现在梳理一下

如果提前把阻塞和转发的大框架定好，那么等到了上机其实就不用再纠结这些问题了。只需要根据题目把条件写好就行。但是如果连扩展代码条件的地方都出错那就是没救了

遵循以下几个步骤：

  1. macro宏文件里面添加指令匹配，同时注意归类的信号也要加上，写31号寄存器的话要加ra信号。

  2. controller必须从头到尾过一遍，有要比较的务必注意在com_op里面补充对应的op。

  3. 跳转信号的话如果要清空延迟槽就得在外部多加一个delay_flush信号，注意该信号和condition是一起的。

  4. 条件访存一堆乱七八糟的需要注意在最后根据输出的condition进行一下目标寄存器和写入数据的修正。

  5. 如果是不符合条件就不写寄存器的，可以把要写入的寄存器设置成0号寄存器。

    把5'b11111改成5'b00000,这样数据通路就不用再重新写了。

如果是branch类条件写的话每一级的RW都需要修改成老样子！！！注意注意！！！

ALUop不可忘改！！

运算，但条件写

可以当成branch类条件写的变种。

在D级的compare直接计算并且判断，然后比较信号往下流水，E级ALU也照常计算。

也即这个条件写是使用RW去控制的。

但凡是条件写，就是有可能不写入，就都需要修改RW！！（除了条件访存）

第二种类型的条件访存，注意添加上这个寄存器不为0号寄存器

因为条件访存令其不写入的方法是把要写的寄存器设置成0寄存器

在捋一遍条件写跳转和条件访存

我的认为是这俩东西不应该有任何的交叉，写条件写跳转就是条件写跳转，条件访存就是条件访存

  1. 条件写跳转的条件写就根据的是D级的compare模块输出的条件决定写还是不写，所以会把compare信号往下传。修改每一级的RW也是因为条件写跳转（RW_W不用改了，在前面改过了），跟条件访存没有任何关系。也正因为如此冒险单元里面才会在地址判断的时候加上RW，判断究竟会不会改变这个寄存器，进而决定了是否会进行转发或者暂停。条件写的regDst输出的就是条件成立的regDst，是否写入是由WE控制的。
  2. 条件访存的写还是不写根据的是W级的compare模块，不写寄存器的话RW为0和写5‘b00000寄存器是一个效果，所以务必在冒险单元上添加上那个不是5’b00000，这样就不会阻塞了，跟RW没有任何关系，不要管

$signed要全加，包括0

比如和0大小比较：

$signed(srcA) > $signed(0)

也就是在compare模块中，如果是有符号比较，最好全部加上$signed符号

对于冒险单元来说，不满足条件不写入改RW和写0号寄存器的作用是一样的

beq类条件写务必在上机的时候添加上每一级的RW，如果写入的寄存器也有不一样的话也需要改变写入的寄存器，也是每一级都要改，compare信号流水，写入数据也是每一级都要改。

冒险单元第三种情况写错了注意！useReg全是D级的！！！

注意aluop不要写错，sll已经占了一个，再往下编写的时候注意！！

offset偏移量到底指的什么？

就是中间隔的指令的条数，比如offset等于2，就是跳转到跟beq之间隔了两条指令的地方。

新的一个点：lhogez不要归入ra，因为那样的话输出的writeReg就是31号寄存器了，但是要求的是writereg输出的应该是条件成立的寄存器。最后的写入数据和寄存器都是经过特判的，所以可以直接去掉，否则M，E级的都得重新改一遍，很麻烦。不归入ra的话自动就是rt寄存器了。

对于D2R和RD这两个信号，都采取在ra后面并上新信号的形式，不容易出错。
