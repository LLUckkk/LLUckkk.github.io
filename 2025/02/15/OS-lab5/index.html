Lab5 上机前复习

最后一次os上机，善始善终！

一些注意事项

  • msyscall括号里注意把参数写完整，不要只写一个系统调用号

  • 磁盘操作中所有的地址操作都需要将物理地址转换成虚拟地址

  • iowrite和ioread的使用注意把va根据len的长度转换为对应类型的指针。

  • 一个磁盘块的大小是4096字节，4KB

  • 目录的最低级的磁盘块中存的是文件控制块的指针

  • FILE2BLOCK：表示一个磁盘块中最多储存多少个文件控制块

  • file_get_block(struct File *file, u_int blockno, void ** blk)指的是获得file文件的第blockno的磁盘块，并将该磁盘块的指针写入blk指向的区域。

  • 如何遍历一个目录中的所有的文件控制块：

    • 先遍历磁盘块，在遍历磁盘块下的文件控制块

    • 使用目录文件大小除以磁盘块的大小得到磁盘块数量

    • 循环内部首先使用try(file_get_block(dir, i, &blk));找到dir中第i个磁盘块，并使blk指向这个磁盘块；

    • 又因为这个磁盘块里面存储的都是文件控制块，所以将其强转为文件控制块指针，指向的是磁盘块里的第一个文件控制块

    • 接着遍历即可

  • write_block：将缓存上修改的值写回磁盘

  • 用户程序在发出文件系统操作请求时，将请求的内容放在对应的结构体中进行消息的传递。这些结构体的定义可在user/include/fsreq.h中查看

  • 文件的偏移指针：fd_offset

外设控制

  • cpu通过读写设备控制器上的寄存器（注意不是cpu寄存器）实现对设备的控制和通信。

  • MIPS体系下，使用MMIO，即内存映射IO机制访问设备寄存器。

    MMIO使用不同的物理内存地址为设备寄存器编址，将一部分对物理内存的访问“重定向”到设备地址空间中。也就是说，CPU对这部分物理内存的访问，等同于对相应设备的访问

  • 对设备物理内存的访问通过kseg1段，因为对该段进行读写不经过TLB映射，且不使用高速缓存。同时，I/O设备的物理地址完全固定，因此我们可以通过简单地读写某些固定的内核虚拟地址来实现驱动程序的功能。

  • 物理地址转换为kseg1段的内核虚拟地址的方法：va = pa + 0xA0000000

  • 不使用缓存的原因是：设备物理内存处的数据不只由CPU决定，还和对应的外设的行为有关。但缓存只记录CPU的读写结果，无法在外设对数据进行修改的时候及时调整。

  • 已知对设备物理内存的访问通过kseg1段，而kseg1段只能在内核态读写，因此需要操作系统提供系统调用来实现在用户态读写设备。也即:

    • sys_write_dev：将起始虚拟地址为va，长度为len的一段数据写到起始物理地址为pa，长度为len字节的物理空间上。

    • sys_read_dev：将起始物理地址为pa，长度为len字节的物理空间中的数据拷贝到起始虚拟地址为va，程度为len字节的空间中。

    • 以上两个函数中，pa对应的都是外设的物理地址，要注意物理地址向内核虚拟地址的转换，因此write就是将内存va中的数据写入pa外设，read即为将外设中的数据读入va的虚拟地址中。

    • 鉴于是对设备的读写，因此函数中要检查pa的合法性，除此之外，len的长度必须是1，2，4

    • 这里使用了ioread和iowrite

      在使用的时候，注意根据len的长度对va进行对应的指针类型转换。写的时候va在参数，读的时候va是等式的一侧，都要进行指针类型的转换。

  • 以上两个系统调用提供的是通用的设备控制接口，针对于IDE磁盘这一特定设备，还需要专门为其编写用户态的控制接口，也即设备驱动程序，设备驱动为操作系统提供了标准化的设备控制接口。

  • IDE外设一般不能立即完成数据操作，需要cpu检查IDE状态并等待操作完成：wait_ide_ready

  • void ide_read(u_int diskno, u_int secno, void *dst, u_int nsecs);
void ide_write(u_int diskno, u_int secno, void *src, u_int nsecs);

使用位图法Bitmap来管理空闲的磁盘资源，用一个二进制位bit标识磁盘中的一个磁盘块的使用情况。

一个FILE结构体的大小为固定的256字节，BY2FILE宏定义

  • 注意目录的文件内容，也就是其磁盘块里存储的内容，是其目录下的文件的文件控制块。

块缓存机制

文件系统服务进程会将一段虚拟地址空间0x10000000-0x4fffffff作为缓冲区以映射磁盘数据，比如，编号为0的磁盘块映射到虚拟地址空间0x10000000-0x10000fff这4096字节大小的空间，其他的依次顺延。

这样，当其他进程想要访问文件数据的时候，只需要将自身虚拟地址的一部分映射到文件系统服务进程缓冲区的相应位置即可。

实现磁盘块与文件系统服务进程内存空间的映射，需要用到三个函数：

  • diskaddr函数，用于将数据块编号转换为缓冲区范围内的对应虚拟地址

  • map_block函数用于分配映射磁盘块需要的物理页面；

    • 先使用block_is_mapped查看blockno对应的磁盘块是否已经被映射到内存，如果是则直接返回0（？or 1）
    • 若没有建立映射，则用diskaddr获得blockno对应的虚拟地址后，调用syscall_mem_alloc函数为该虚拟地址分配一页物理页面
    • map_block所对应的数据块在之前未被使用，其中的数据是无效数据，因此无需写回。

  • unmap_block函数用于释放用来映射磁盘块的物理页面。

    • 使用block_is_mapped获取对应磁盘块的虚拟起始地址
    • 如果磁盘块非空闲且磁盘块缓存被写过（也即其被映射到的内存区域写过），调用write_back函数将磁盘块缓存的数据写回磁盘
    • 最后调用syscall_mem_unmap函数解除对应磁盘块缓存虚拟页面到物理页面的映射。

文件操作

fsipc

MOS的文件系统服务由用户进程提供，因此为了实现创建文件、读写文件等等文件操作，就需要使用进程间通信机制。

请求文件系统服务进程的接口为fsipc。

不同的操作类型通过参数type区分，并被进一步封装为不同的函数。

fsreq为对应请求的结构体，dstva，perm一般为0？

文件系统服务进程的入口在fs/serv.c中，程序主体为死循环，循环中调用ipc_recv不断接受其他进程发来的请求，并通过serve_table转发到不同的处理函数，处理结束后，调用ipc_send返回调用的结果。

file.c

在fsipc函数的基础上继续进行封装以提供更加清晰有好的文件操作接口

  • open函数，打开指定文件，返回文件描述符

    • 先使用fd_alloc和fsipc_open函数分配一个文件描述符

    • 之后遍历文件内容，使用fsipc_map函数为文件内容分配页面并映射
    • 最后使用fd2num函数返回文件描述符

  • read函数

    • 读取一个文件描述符编号fdnum，一个缓冲区指针buf，和一个最大读取字节数n作为输入参数，返回成功读取的字节数并更新文件描述符的偏移量

    • 调用设备的读取函数dev_read从文件当前的偏移位置读取数据到缓冲区
    • 若读取操作成功，函数更新文件描述符的偏移量并返回成功读取的字节数。

  • remove函数

    remove fsipc_remove file_remove 调用链

    • 将缓冲区fsipcbuf 视为一个Fsreq_remove结构体，代表将要发送的是remove请求，并将路径字符串复制到结构体的路径字段中。 ？？？？

    serve_remove是文件删除请求的处理函数，调用的file_remove函数实现了文件的删除。

文件描述符

文件描述符是系统提供给用户的整数，用于在描述符表中进行索引。

  • 使用open在描述符表的指定位置存放被打开文件的信息
  • 使用close将描述符表中指定位置的文件信息释放
  • write和read修改描述符表中指定位置的文件信息

open函数的返回值为整数，称为文件描述符，其本质是对描述符表的索引，也就是说，每个文件描述符对应着一个Filefd结构体

该结构体在文件系统服务进程中创建，并通过ipc_send共享给其他进程

  • 令子进程和父进程的偏移指针独立不再继承

    注意到父进程的Fd的映射含有PTE_LIBRARY，因此fork时不会对Fd进行COW保护，因此，Fd对于父子进程时共享的，一方修改，另一方也能读到。

    也就是说，单凭修改子进程的Fd实现该功能是错误的，此时父子进程的偏移都会改变

    因此，我们需要在ffd映射给用户进程时，检验模式，如果是该模式，则不加PTE_LIBRARY权限。则此时fd会被COW保护，也即相互独立了
